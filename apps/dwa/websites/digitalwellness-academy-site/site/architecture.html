PLATFORM ARCHITECTURE

The architecture behind a HIPAA-compliant, two-school learning platform

Two course schools on one shared identity layer. A custom distress classifier trained for mental-health context. A provider coordination portal that exposes outcomes without exposing PHI. And a data pipeline that turns every lesson interaction into a signal to improve the next cohort of learners.

System-level view

Four tiers, each designed for a different class of risk and latency.

Tier 1 · Learner Layer

Next.js 16 app on Node

Server-rendered learner and provider interfaces. Lucia session-based auth. Routes segmented into /academy/* (therapeutic) and /optimization/* (five pillars) with shared identity but isolated progress tables.

  • • Lucia auth + session cookies (no JWT sprawl)
  • • Server components for lesson content
  • • Client components for interactive widgets
  • • Tailwind 4 + MDX for lesson authoring
Tier 2 · Data Layer

Postgres + Redis + Drizzle

Postgres is the durable store (18 tables: users, enrollments, progress, assessments, quiz responses, mood tracking, thought records, distress events, provider relationships, forum posts). Redis handles sessions, rate limiting, and the distress-classifier request queue.

  • • Drizzle ORM with typed schema
  • • Row-level security via provider relationships
  • • Encryption at rest; TLS in transit
  • • Audit log on every PHI-adjacent write
Tier 3 · Safety AI

DistilBERT distress classifier

A dedicated FastAPI service hosts a DistilBERT model fine-tuned on a mental-health distress corpus. The learner app calls this service synchronously at every text-input boundary - journals, thought records, forum posts, assessment free-text, onboarding reflections.

  • • 66M-parameter DistilBERT base
  • • Three-level output: crisis / mild / none
  • • Zero-knowledge: text discarded after inference
  • • Only {level, confidence, ts} persisted
Tier 4 · Analytics

Metabase + adaptive-learning pipeline

Every lesson view, component interaction, quiz response, and assessment score emits structured events into Postgres. Metabase dashboards surface cohort-level outcomes to the practice. De-identified aggregates feed the adaptive-learning model training runs.

  • • Cohort outcome dashboards (practice-scoped)
  • • Lesson feedback + quiz performance heatmaps
  • • Training corpus for course sequencing models
  • • Drop-off risk signals back to provider portal

Two-school isolation

Shared auth, isolated progress. A therapeutic enrollment never bleeds into optimization analytics, and vice versa.

Therapeutic School · /academy/*

24 courses · 217 lessons

Clinical content with symptom-driven routing. Lessons are authored in .md. Enrollments carry a care_context field flagging provider-referred entry. Progress rolls up into PHQ-9/GAD-7 trajectory charts visible to the assigned provider.

  • →Symptom-based onboarding assessment drives course recommendations
  • →Thought records, exposure hierarchies, body maps
  • →All text inputs pass through the distress classifier
  • →Outcomes dashboard visible only to that patient's provider
Optimization School · /optimization/*

19 courses · 375 lessons

Five-pillar peak-performance content. Lessons are authored in .mdx with embedded interactive components. No clinical assessment gating - anyone can enroll. Progress stays inside optimization tables; nothing surfaces to providers unless the user explicitly opts in.

  • →Five Pillars: Physical Vitality, Social Connection, Mental Clarity, Emotional Resilience, Purpose
  • →Keyword-gate safety layer (not DistilBERT) on public inputs
  • →Public-facing SEO lessons for licensee content marketing
  • →No provider coordination by default

Why shared auth but isolated progress?

A patient with panic disorder might also train for a marathon. The platform lets them do both without their wellness-seeking optimization data feeding into the clinical record their provider reviews. Isolation is enforced at the database row level - not by application convention - so a developer mistake can't leak optimization activity into therapeutic analytics.

SAFETY AI

DistilBERT distress classifier

A domain-specific model wins over a generic LLM here. Smaller, faster, cheaper per call, and trained on the exact language of mental-health distress.

How a text input flows through it

  1. 1Learner submits a thought record, journal entry, or forum post.
  2. 2Learner app POSTs text to /api/safety/classify.
  3. 3API route forwards to FastAPI service; text is held in memory only during inference.
  4. 4Model returns {level, confidence}; text is immediately discarded.
  5. 5If crisis: 988 modal + provider alert + audit log entry.
  6. 6If mild: gentle check-in prompt; metadata logged.
  7. 7If none: user continues. Only an event counter increments.

Why not use a big LLM?

Latency. DistilBERT inference on a T4/P100 GPU runs in the 50–200ms range. A hosted LLM API adds hundreds of milliseconds plus variance. The classifier sits on the critical path of every thought record save - it cannot be slow.

Cost. A fine-tuned 66M-param model on our own GPU costs a fraction of a cent per inference. Running every learner utterance through a frontier LLM would be an order of magnitude more expensive, for worse domain accuracy.

Predictability. A classifier emits a bounded output: three labels + a confidence score. An LLM emits free text that you then have to classify again. For a safety-critical path, bounded beats generative.

Privacy posture. A model we host, we control. Learner text never crosses a third-party API boundary.

HIPAA posture & privacy stance

What we collect, what we don't, who sees it, and under what agreement.

⚠
Your hosting provider must sign a BAA

The platform code is designed for HIPAA compliance — but compliance doesn't activate until your cloud hosting provider has signed a Business Associate Agreement with your practice. AWS, Google Cloud, and Azure will sign one. Hostinger, Vercel (non-enterprise), and shared hosting providers won't.

Full BAA provider guide + portability docs →

What we store

  • ✓Structured responses: quiz answers, PHQ-9/GAD-7/PCL-5 scores, mood scale entries
  • ✓Progress metadata: lessons viewed, course completion, time-on-task
  • ✓Distress events as {level, confidence, timestamp, course, lesson}
  • ✓Provider↔patient relationship records with audit trail
  • ✓Feedback ratings on lessons (not verbatim text unless the user chooses to submit it)

What we never store

  • ✗The text of thought records, journal entries, or forum posts after classification
  • ✗Audio or video recordings
  • ✗Any data sent to a third-party LLM (we don't send any)
  • ✗PHI in analytics aggregates - only de-identified rollups for the adaptive-learning pipeline
  • ✗Cross-practice data visibility - each practice sees only their own patients

Infrastructure posture

Hosting

Designed for deployment on BAA-compliant cloud (GCP, AWS, Azure). Staging on Dokploy-managed infrastructure. Production migration to a BAA-signed provider is a prerequisite for each licensee launch.

Access control

Provider accounts scoped to their practice. Patient rows keyed to provider relationship. Audit log captures every PHI-adjacent read and write. Secrets rotate per environment; no shared credentials between practices.

Breach posture

Because we never store learner text, the blast radius of a database leak is structured metadata, not reflective personal writing. That's a deliberate design choice - not a side effect.

Adaptive-learning data pipeline

Every lesson, quiz, and assessment emits a signal. Those signals become training data. That's the moat.

1
Emit
Lesson view, component interaction, quiz answer, assessment submission
2
Persist
Structured events land in Postgres with user_id + course_id + lesson_id keys
3
Aggregate
De-identified rollups feed cohort dashboards and training datasets
4
Improve
Flagged lessons rewritten; course sequencing retrained; provider dashboards updated

Quiz-performance heatmap

Which questions do most learners get wrong on the first try? That's a signal the lesson didn't teach the concept clearly enough - flag it for rewrite.

Drop-off signals

Which lesson in which course has the highest abandonment rate? Flag for restructuring. Learners who drop there rarely come back.

Outcome correlation

Pre/post PHQ-9 deltas mapped to course sequences. Which content orderings produce the biggest clinical improvements? Feed back into onboarding recommendations.

Provider coordination layer

What a clinician sees, what they don't, and how a crisis alert actually arrives.

Patient roster view

Provider logs into /provider/*. The dashboard is scoped to their assigned patients only, verified at the DB row level - not application layer.

  • • Trend lines on PHQ-9, GAD-7 (when taken)
  • • 30-day mood rolling average
  • • Course activity summary (no lesson text)
  • • Active distress alerts at the top, ranked by severity
  • • Acknowledge/resolve workflow with audit trail

Crisis alert path

When the classifier fires at crisis level:

  1. 1. Learner immediately sees a 988 resource modal.
  2. 2. Assigned provider receives email + in-app notification.
  3. 3. Alert payload contains: course, lesson, timestamp, confidence - no learner text.
  4. 4. Alert appears in the provider's roster until acknowledged.
  5. 5. Acknowledgment writes to the audit log: who, when, what action.
